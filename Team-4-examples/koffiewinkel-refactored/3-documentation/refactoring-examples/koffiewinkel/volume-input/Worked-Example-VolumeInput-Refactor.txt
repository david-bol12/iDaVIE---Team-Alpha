Worked Example 2: VolumeInputController Refactor (Koffiewinkel)
Author: Colin Forde
Team evidence: CATME peer evaluation, 22 May 2026 (second worked example)

================================================================================
1) PROBLEM STATEMENT (BEFORE)
================================================================================

Target: VolumeInputController.cs (~1635 lines) — interaction-system hub for VR.

Baseline issues:
- God class: SteamVR wiring + locomotion + interaction FSM + painting + cursor UI text + vignette + quick menu attachment in one file
- Two state machines in one class: LocomotionState (enum + switch in Update) and InteractionStateMachine (Stateless)
- Low testability: direct SteamVR types, scene Find calls, Camera.main in teleport/video paths
- High coupling to menus, voice commands, and feature anchors via public surface
- Known bug: UpdateVignette applied VignetteIntensity twice per frame (second loop overwrote conditional logic)

ISO/IEC 25010: poor modularity, analyzability, and modifiability on a critical-path class.

OUT OF SCOPE FOR THIS EXAMPLE:
- VolumeDataSetRenderer, shader/cube rendering, MomentMap rendering (Rendering sub-team)
- Refactoring Menu/ UI controllers themselves

================================================================================
2) REFACTORING OBJECTIVE
================================================================================

Split VolumeInputController so it only:
- Owns SteamVR action listeners and hand/VR-family setup
- Routes input events to collaborators
- Exposes a stable public API for VolumeCommandController, PaintMenuController, QuickMenuController, etc.

Move domain behaviour into six collaborators under Assets/Scripts/Interaction/.

================================================================================
3) DESIGN APPROACH
================================================================================

Patterns:
1. Facade — VolumeInputController remains the Unity entry point
2. Strategy — ILocomotionController encapsulates move/scale/threshold/Z modes
3. State — IInteractionController owns Stateless FSM for paint/select/edit/video
4. Pure helper — ICursorInfoFormatter (no MonoBehaviour, unit-testable strings)
5. Adapter — IGaze / CameraGazeProvider replaces direct Camera.main for head-based actions

================================================================================
4) IMPLEMENTATION CHANGES
================================================================================

4.1 VolumeInputController (thin router)

Keeps:
- SteamVR fields: _grabGripAction, _grabPinchAction, _quickMenuAction
- OnEnable/OnDisable listener registration + hand transform setup + DetermineVRFamily()
- All OnXxx handlers (OnGripChanged, OnPinchChanged, OnQuickMenuChanged, menu buttons, etc.)
- PrimaryHand, PushToTalk events, compatibility wrappers (StartThresholdEditing, InteractionStateMachine proxy)

Update() becomes:
  _locomotionController?.Update(Time.deltaTime);
  _vignetteController?.Update(Time.deltaTime);

OnGripChanged routes grip count to locomotion:
  case 0 -> TransitionToIdle()
  case 1 -> TransitionToMoving()
  case 2 -> TransitionToScaling()

OnPinchChanged routes to _interactionController.Fire(InteractionStarted/Ended) except video-record cooldown path.

4.2 ILocomotionController / LocomotionController

Extracted:
- LocomotionState enum (Idle, Moving, Scaling, EditingThresholdMin/Max, EditingZAxis)
- UpdateMoving, UpdateScaling (yaw/roll axis locking), UpdateEditingThreshold, UpdateEditingZAxis
- Grip transition helpers, line renderers for scale visualization
- StartThresholdEditing, StartZAxisEditing, EndEditing

4.3 IInteractionController / InteractionController

Extracted:
- Interaction states/events, Stateless configuration, selection/region edit, paint mode entry hooks
- Update() when locomotion is Idle: cursor position, painting, region bounds, hand info text
- SetHoveredFeature / ClearHoveredFeature for feature anchors

4.4 IBrushController / BrushController

Extracted:
- BrushSize, AdditiveBrush, SourceId, increase/decrease/reset, undo/redo, AddNewSource, StartEditSourceID

4.5 IVignetteController / VignetteController

Extracted:
- Vignette fade logic; FIXED duplicate dataset loop (single pass sets VignetteIntensity)

4.6 ICursorInfoFormatter / CursorInfoFormatter

Pure C#:
- FormatCursorInfo, FormatSelectionInfo, FormatAngle
- No UnityEngine dependency beyond data types on VolumeDataSetRenderer

4.7 IQuickMenuPositioner / QuickMenuPositioner

Extracted:
- Show(handIndex, handTransform), Hide() for CanvassQuickMenu attachment

4.8 IGazeProvider / IGaze / CameraGazeProvider

Cross-team contract (Sub-team 3 rendering):
  IGazeProvider — GazeDirection, GazeFocusPoint (Vector2 screen [0,1]), IsGazeAvailable

Interaction extension (IGaze : IGazeProvider):
  GazeOrigin, GazeRotation, GazeConfidence, IsTracking, GazeRay — VolumeInputController teleport/video

CameraGazeProvider implements IGaze; GazeFocusPoint from WorldToViewportPoint; unavailable → (0.5, 0.5).
VolumeInputController no longer calls Camera.main directly for teleport or video head mode.

================================================================================
5) BEHAVIOURAL COMPATIBILITY
================================================================================

Preserved:
- public InteractionState / InteractionEvents enums on VolumeInputController
- InteractionStateMachine via InteractionStateMachineProxy (menus still Fire(...) )
- Voice-driven calls: StartThresholdEditing, EndEditing, SetBrushAdditive, AddNewLocation, etc.
- External callers: PaintMenuController, QuickMenuController, VideoRecordMenuController, VolumeCommandController, FeatureAnchor hover

Not changed:
- Rendering behaviour inside VolumeDataSetRenderer
- Menu layout or mask/crop semantics

================================================================================
6) QUALITY IMPACT (BRIEF / CK)
================================================================================

Expected improvements:
- Understandability: each collaborator has one concern; router ~620 lines vs ~1635
- Analyzability: locomotion vs interaction vs brush visible in separate files
- Modifiability: new locomotion mode or interaction transition localized
- Testability: CursorInfoFormatter testable without Unity; collaborators mockable behind interfaces

Trade-off: more types (interfaces + implementations); CBO spread across Interaction assembly.

================================================================================
7) VERIFICATION PLAN
================================================================================

1. Play mode: single-grip move, two-grip scale/rotate, release to idle
2. Voice/menu: edit min/max threshold, save threshold (via VolumeCommandController)
3. Paint mode: brush resize (menu up/down), paint stroke, undo/redo (secondary hand)
4. Quick menu: open on non-primary hand quick-menu button
5. Interaction FSM: region create, feature edit anchors, paint mode from menu
6. Video record mode: record position with cooldown
7. Teleport / saved video location restore (gaze-based head mode)
8. NDepend: compare VolumeInputController WMC/RFC before vs after on same build

================================================================================
8) RISKS AND LIMITATIONS
================================================================================

- UpdateDataSets still uses GameObject.Find("VolumeDataSetManager") — future DatasetRegistry injection
- TakePicture still finds CameraController in scene
- EnterPaintModeCore / mask init remain on router (small domain leak; candidate for next slice)
- FeatureAnchor still uses FindObjectOfType<VolumeInputController> for hover (pre-existing)
